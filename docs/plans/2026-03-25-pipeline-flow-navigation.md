# Pipeline Flow & Navigation Plan

> **For Claude:** REQUIRED SUB-SKILL: Use superpowers-extended-cc:executing-plans to implement this plan task-by-task.

**Goal:** Clean up pipeline controls, navigation, and state transitions. Make it intuitive. Eliminate dead-end navigation. Add restart capability.

**Architecture decision (from frontend audit):** Use a shared `layout.tsx` with persistent device header + route-based nav links — NOT single-page tabs. 2. **Dead-end navigation** — Editor says "manifest not found", user clicks back, lands on wrong page. Preview says "not generated yet", no clear path forward.

## PHASE 1: Restart Pipeline (CRITICAL — do first)

### Task 1: Add "Restart Pipeline" API + UI

Outdated pipelines need to be re-run with the latest SOUL updates, codegen fixes, and validator improvements. This is the most important missing feature.

**What restart does:**
- Kills running process AND child agent process (if any, handle stale PIDs)
- Removes worktree (`.worktrees/{id}/`)
- Preserves contractor work and device identity
- Wipes pipeline execution state
- Re-creates initial state from device metadata
- Auto-starts the runner from preflight

**Explicit file preservation/deletion list:**

| Path | Action | Reason |
|---|---|---|
| `.pipeline/{id}/manifest-editor.json` | PRESERVE (backup to saved/) | Contractor positions are SACRED |
| `.pipeline/{id}/manifest.json` | DELETE | Will be regenerated by gatekeeper |
| `.pipeline/{id}/state.json` | DELETE + RECREATE | Fresh initial state |
| `.pipeline/{id}/templates.json` | DELETE | Will be regenerated by layout engine |
| `.pipeline/{id}/inferred-layout.json` | DELETE | From old editor inference |
| `.pipeline/{id}/agents/` | DELETE (entire directory) | All agent checkpoints/outputs |
| `.pipeline/{id}/cost.json` | DELETE | Old cost data |
| `.pipeline/{id}/runner.log` | DELETE | Old logs |
| `.pipeline/{id}/backups/` | DELETE | Old manifest backups |
| `.pipeline/{id}/input/manuals/` | PRESERVE | Already downloaded PDFs, expensive to re-fetch |
| `.pipeline/{id}/input/photos/` | PRESERVE | Already downloaded photos |
| `.worktrees/{id}/` | DELETE | Old git worktree, will be recreated |
| `src/components/devices/{id}/` | PRESERVE | Generated panel code, will be overwritten by next codegen |

**API:** `POST /api/pipeline/[deviceId]/restart`

```typescript
// 1. Read existing state for device metadata (manufacturer, deviceName, manualPaths, budgetCapUsd)
// 2. Kill runner PID if alive (handle stale: check isProcessAlive first)
// 3. Kill child PID if alive
// 4. Backup manifest-editor.json to .pipeline/saved/{id}/
// 5. Delete: state.json, manifest.json, templates.json, inferred-layout.json, agents/, cost.json, runner.log, backups/
// 6. Remove worktree: .worktrees/{id}/
// 7. Create fresh state via createInitialState({ deviceId, deviceName, manufacturer, manualPaths, budgetCapUsd })
// 8. Set state.codegenCompleted = false (new field — see Task 7)
// 9. Spawn runner
// 10. Return { status: 'running', pid }
```

**UI:** "Restart" button in the pipeline detail header.
- Visible when: `status === 'paused' || 'completed' || 'failed'`
- Confirmation dialog: "This will re-run the entire pipeline with the latest improvements. Your editor positions will be preserved."
- Not visible when running (must pause first)

**Files:**
- Create: `src/app/api/pipeline/[deviceId]/restart/route.ts`
- Modify: `src/app/admin/[deviceId]/page.tsx` — add Restart button
- Modify: `src/lib/pipeline/state-machine.ts` — add helper to create restart state from existing metadata

### Task 2: Add "Reset to Phase" in diagnostics

Sometimes you just want to re-run the gatekeeper, not the whole pipeline.

**API:** Extend `POST /api/pipeline/[deviceId]/recover` with `action: 'reset-to-phase'` and `targetPhase` parameter.

**Data preservation semantics:** Reset to phase N means: keep all outputs from phases 0 to N-1, delete N onwards.
- Reset to gatekeeper: keeps parser output (spatial-blueprint.json), deletes gatekeeper manifest + everything after
- Reset to parser: keeps preflight (manuals/photos), deletes parser output + everything after
- Reset to preflight: deletes everything except device metadata (like restart but preserves manifest-editor.json)

**Validation:**
- `targetPhase` must be in `PHASE_ORDER` array
- Cannot jump forward (only backward)
- Cannot reset past an unresolved escalation
- Must pause the pipeline first (status must be paused/completed/failed)

**UI:** Dropdown in diagnostics panel: "Reset to: [Preflight | Parser | Gatekeeper | Layout Engine]"

**Files:**
- Modify: `src/app/api/pipeline/[deviceId]/recover/route.ts`
- Modify: diagnostics component in `PipelineDetail.tsx`

## PHASE 3: Status-Aware Controls

### Task 6: Action button visibility map

Implement a single source of truth for which buttons are visible/enabled:

```
pending:    [Cancel]
running:    [Pause]
paused:     [Resume, Cancel, Restart]
completed:  [Restart]
failed:     [Restart]
```

**Pause** (new): Sends SIGTERM to runner + child agent. Sets status to paused. Preserves ALL state. Pipeline can be resumed from where it left off.
**Cancel**: Sets status to failed, cleans up worktree. Confirmation dialog: "This will permanently stop the pipeline. Use Pause to resume later."
**Restart**: Wipes execution state, preserves editor positions. Confirmation dialog: "Re-run entire pipeline with latest improvements. Editor positions preserved."

**Pause API implementation:**
```typescript
case 'pause': {
  // Kill runner gracefully
  if (state.runnerPid && isProcessAlive(state.runnerPid)) {
    process.kill(state.runnerPid, 'SIGTERM');
  }
  // Kill child agent process
  if (state.childPid && isProcessAlive(state.childPid)) {
    process.kill(state.childPid, 'SIGTERM');
  }
  state.status = 'paused';
  state.runnerPid = null;
  state.childPid = null;
  writeState(deviceId, state);
  return NextResponse.json({ status: 'paused', action: 'pause' });
}
```

**Difference from `fix-stale`:** `fix-stale` only applies when the process is already dead but status says running. `pause` actively kills a running process.

**Files:**
- Modify: `src/components/admin/DeviceHeader.tsx` — implement status map
- Modify: `src/app/api/pipeline/[deviceId]/recover/route.ts` — add `pause` action

### Task 7: Nav link gating logic

The DeviceNav component checks pipeline state to enable/disable route links:

```typescript
const gatekeeperPassed = pipeline.phases.some(p => p.phase === 'phase-0-gatekeeper' && p.status === 'passed');
const codegenCompleted = pipeline.codegenCompleted === true; // Runtime boolean, NOT compile-time DEVICE_REGISTRY

// Editor: enabled if gatekeeperPassed
// Preview: enabled if codegenCompleted
```

**Why not `DEVICE_REGISTRY[deviceId]`:** The device registry is a compile-time static import. When codegen runs at runtime, the registry doesn't update until server restart. `codegenCompleted` in state.json is a runtime-safe check.

**Where `codegenCompleted` gets set:**
- Codegen API sets `state.codegenCompleted = true` after successful run
- Restart API resets it to `false`
- `createInitialState` initializes as `false`

Disabled links render as `<span>` (not `<a>`) with `cursor-not-allowed`, `opacity-50`, and tooltip on hover.

**Files:**
- Modify: `src/components/admin/DeviceNav.tsx` — gating logic + tooltips
- Modify: `src/lib/pipeline/state-machine.ts` — add `codegenCompleted` to PipelineState
- Modify: `src/app/api/pipeline/[deviceId]/codegen/route.ts` — set `codegenCompleted = true`
- Modify: `src/lib/deviceRegistry.ts` — export `hasDevice(id: string): boolean`

## Frontend Architecture Decision (from component architect review)

**Shared layout + route links** over single-page tabs because:
- Editor, Overview, and Preview have incompatible height models (full-viewport canvas vs scrollable document vs static render)
- Forcing all three into one tab content area requires conditional CSS that breaks when a new tab is added
- Route-level layouts give each view its own height contract while sharing the chrome
- SSE connection moves to the layout so it persists across route switches
- Next.js App Router code-splits per route — editor bundle only loads when navigating to `/editor`
- Deep-linking works naturally — contractor bookmark `/admin/cdj-3000/editor` just works
